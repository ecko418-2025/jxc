# 酒店保洁产品进销存系统 — 技术改进路线图

本文档基于对当前 `v1.0.0` 版本源码的分析，制定了从“内存/本地驱动”向“工业级云端 SaaS 架构”转型的技术改进计划。

---

## 1. 核心架构重构：云端优先 (Cloud-First)

**现状：** 数据先写内存缓存再异步同步，缺乏写入确认和冲突处理。

### 改进任务：
- [ ] **移除内存同步逻辑**：将 `db.ts` 中的 `cache` 对象和 `syncToCloud` 逻辑替换为直接调用 CloudBase SDK 的 Promise/Async 模式。
- [ ] **引入 React Query (TanStack Query)**：
  - 使用 `useQuery` 管理产品、订单列表的获取，利用其内置的缓存、自动刷新和 Loading 状态。
  - 使用 `useMutation` 管理 CRUD 操作，确保在云端成功返回后再更新 UI。
- [ ] **主键策略调整**：弃用 `generateId()`，统一使用云数据库自动生成的 `_id`。

---

## 2. 数据可靠性：云端事务化 (Transactions)

**现状：** 库存变动与订单状态更新在前端分步执行，存在“断网导致数据不一致”的风险。

### 改进任务：
- [ ] **核心业务逻辑云函数化**：
  - 创建 `confirmOrder` 云函数：原子性执行“修改订单状态 + 增减库存 + 记录库存流水”。
  - 创建 `initSystem` 云函数：安全执行演示数据的初始化，避免重复生成。
- [ ] **引入数据库事务**：在云函数中使用 `db.runTransaction()`，确保“进-销-存”数据链条的绝对完整。

---

## 3. 系统安全：权限与校验 (Security)

**现状：** 集合 ACL 为 `PUBLIC`，前端计算逻辑易被篡改。

### 改进任务：
- [ ] **配置安全规则 (Security Rules)**：
  - 在 `cloudbaserc.json` 中定义细粒度规则：
    - `inventoryLogs`：仅允许通过云函数写入（只读）。
    - `orders`：仅登录用户可读写，且禁止修改已完成订单的关键字段。
- [ ] **敏感计算后端化**：订单总金额、折扣计算由云函数校验，防止前端传递恶意修改后的价格。

---

## 4. 前端工程化：模块化与性能 (Frontend Refactoring)

**现状：** `db.ts` 文件过于庞大，业务逻辑与数据访问耦合。

### 改进任务：
- [ ] **服务层拆分 (Service Pattern)**：
  ```bash
  src/services/
  ├── productService.ts    # 产品相关 API
  ├── orderService.ts      # 订单业务流
  └── inventoryService.ts   # 库存统计与预警
  ```
- [ ] **Ant Design 组件优化**：
  - 引入 `App` 包裹组件，使用命令式 `message` 和 `modal`。
  - 优化大表格性能，开启虚拟列表（Virtual List）应对上千条产品记录。
- [ ] **全局状态管理**：使用 `Zustand` 替代 `db.ts` 中的全局变量，存储当前用户信息和系统配置。

---

## 5. 监控与运维 (Observability)

**现状：** 缺乏错误日志，难以排查线上问题。

### 改进任务：
- [ ] **统一错误处理**：封装全局 `request` 工具，拦截云端异常并展示友好提示。
- [ ] **库存审计增强**：在 `inventoryLog` 中增加 `requestId`，可追溯单次操作关联的所有库存变动。

---

## 6. 实施优先级划分

### 第一阶段 (P0 - 关键修复)
1. 将 `confirmReceipt` 和 `confirmShipment` 迁移至云函数。
2. 修复 `db.ts` 的 ID 生成逻辑。
3. 配置基础安全规则。

### 第二阶段 (P1 - 体验与稳定性)
1. 引入 React Query 重构数据获取流。
2. 拆分 `db.ts` 到服务层。
3. 增加数据导出为 Excel 的服务端实现。

### 第三阶段 (P2 - 扩展功能)
1. 增加 ECharts 数据可视化看板。
2. 开发独立的小程序端（利用 CloudBase 一套后端）。
